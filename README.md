# Shortest Path Algorithms on a Real Road Network

実在の道路ネットワーク（OpenStreetMap・横浜市周辺）上で、5種類の最短経路アルゴリズムを **Go で自作実装**し、計算量の理論と実測を突き合わせて比較したプロジェクトです。標準ライブラリのみで実装し、優先度付きキュー（二分ヒープ）も自前で書いています。

大学の課題として作成しましたが、「教科書の擬似コードを実データで動かすと何が効いてくるのか」を定量的に検証することを主眼に置いています。

## 何を比較したか

| アルゴリズム | 講義での扱い | 計算量 | 入力表現 |
|---|---|---|---|
| Dijkstra（隣接行列） | ○ | O(V²) | 隣接行列 |
| Dijkstra（優先度付きキュー） | ○ | O((V+E) log V) | 隣接リスト |
| Bellman-Ford | ○ | O(VE) | 隣接リスト |
| A* | 授業外（発展） | ヒューリスティック依存 | 隣接リスト＋座標 |
| 双方向 Dijkstra | 授業外（発展） | O((V+E) log V) | 隣接リスト |

授業で扱った3種に加え、自主的に A* と双方向 Dijkstra を実装し、計5種を同一データで横並びに評価しました。

## データセット

- **OpenStreetMap** の道路ネットワーク（横浜市周辺、高速＋主要幹線道 `motorway|trunk|primary` に限定）
- Overpass API 経由で取得（[Overpass Turbo](https://overpass-turbo.eu/) を利用）
- 規模: **V = 11,261 ノード、E = 11,343 エッジ**（平均次数 ≈ 2.0 の極めて疎なグラフ）
- 各ノードに緯度経度を持つため、A* のヒューリスティックに地表上の直線距離をそのまま使える
- エッジ重み = 隣接ノード間の直線距離（メートル、正距円筒近似）

> データの著作権について: 道路データは © OpenStreetMap contributors、[Open Database License (ODbL)](https://opendatacommons.org/licenses/odbl/) の下で提供されています。

## 主な結果

代表ペア（始点 → 終点、道路距離 17.9 km、迂回率 0.887）での計測値（実行時間は **100回計測の中央値**）:

| アルゴリズム | 緩和回数 | ノード展開数 | 実行時間（中央値） |
|---|--:|--:|--:|
| Dijkstra（行列） | 16,405 | 8,098 | 125 ms |
| Dijkstra（PQ） | 16,405 | 8,099 | 145 µs |
| Bellman-Ford | 6,329,394 | — | 7.8 ms |
| A* | 6,972 | 3,405 | 162 µs |
| 双方向 Dijkstra | 8,975 | 4,398 | 81 µs |

ここから読み取れたことの一例:

- **同じ Dijkstra でも実装で約900倍**。隣接行列 O(V²) は疎グラフでは無駄が大きく、優先度付きキュー版が圧勝する（125 ms → 145 µs）。理論計算量の差がそのまま現れる。
- **A* の効きはグラフの「迂回率」に強く依存する**。始点-終点を道路距離で層化した10ペアで計測したところ、直線に近い経路（迂回率0.985）では緩和196回まで枝刈りできる一方、大きく迂回する経路（迂回率0.541）では2,144回まで膨らみ、約11倍の開きが出た。**迂回率と緩和削減率には負の相関（r = −0.79）**を確認した。
- **双方向 Dijkstra は探索空間がほぼ半分**になり、このデータでは A* より速い場面が多かった。ただし実行時間の標準偏差が大きく、これは GC のポーズによるスパイクが原因と切り分けた。
- **正しさの検証**: 5アルゴリズムすべてが代表ペアで同一の経路長（17,872.27 m、浮動小数の演算誤差 < 1e-6 m）を返すことを `main.go` の `verifyPathLength` で確認。

## 実装上こだわった点

- **優先度付きキューを自作**（`pqueue.go`）。講義の decrease-key 方式ではなく、緩和成功時に新エントリを push し、pop 時に陳腐化エントリ（`item.dist > dist[id]`）を読み捨てる **重複 push 方式**を採用。計算量は同じ O((V+E) log V) だが、push 回数とヒープサイズが O(E) になるトレードオフを理解した上で選んでいる。
- **A* のヒューリスティックは admissible（過大評価しない）**ことを保証。直線距離は道路距離以下なので最短性が保たれる。`h ≡ 0` なら A* は Dijkstra に一致する点も実装・考察で確認。
- **Bellman-Ford は早期終了**（更新が起きなくなったら打ち切り）を入れ、最悪 O(VE) と実測のズレを説明できるようにした。
- **評価指標を実行時間だけに頼らない**。マシン依存でブレる実行時間に加え、決定的に測れる**緩和回数・ノード展開数・push 回数・最大ヒープサイズ**を主指標に据えた。

## 実行方法

```bash
go run .    # ベンチマークを実行（10ペア層化サンプリング＋代表ペア＋正しさ検証）
```

`data/osm_data.json` に OSM データが同梱されているので、追加の取得作業なしで動きます。

## ディレクトリ構成

```
.
├── main.go            ベンチマーク・始点終点の層化サンプリング・正しさ検証
├── graph.go           Graph 型（隣接リスト＋緯度経度）と距離計算
├── osm.go             OSM JSON → Graph への変換
├── pqueue.go          自作二分ヒープ（最小優先度付きキュー）
├── dijkstra.go        Dijkstra（隣接行列版／優先度付きキュー版）
├── bellmanford.go     Bellman-Ford（早期終了つき）
├── astar.go           A*（直線距離ヒューリスティック）
├── bidijkstra.go      双方向 Dijkstra ＋ 計測用 Metrics 型
└── data/osm_data.json    横浜市周辺の道路ネットワーク（ODbL）
```

## 使用技術

- **言語**: Go（標準ライブラリのみ。外部依存なし）
- **データ**: OpenStreetMap / Overpass API
- **可視化**: matplotlib（散布図・棒グラフ）
